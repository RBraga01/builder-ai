What gets skipped without this pack

Four ways AI features fail in production

Not edge cases. The default outcome when no enforcement exists.

No Eval

"Shipped on vibes"

The feature launched. Nobody ran numbers. Three weeks later, a prompt change shifted accuracy from 89% to 72% and no one noticed until support tickets arrived.

Blocked by eval-before-ship — requires pass rate, baseline, failure analysis before merge
No Versioning

"Which prompt is live?"

The prompt was edited directly in the environment variable. No history, no rollback path. The regression happened three deploys ago and nobody can tell which change caused it.

Blocked by prompt-versioning — semver files, git history, one-line rollback path required
No Fallback

"3am incident"

The API returned a 503. The feature threw an unhandled exception. Users saw a blank screen. The on-call engineer added a try/catch at 3am. The fallback should have been designed before the feature shipped.

Blocked by fallback-required — all 4 failure modes handled before ship
No Safety Review

"Day-two jailbreak"

The system prompt had no injection resistance. A user found the bypass on day two and posted it. The fix took three days. The safety review would have taken two hours.

Blocked by ai-safety-review — prompt injection, hallucination risk, output safety, agentic scope
How enforcement works

A format that cannot be faked

Each skill has a Completion Statement Format — a literal code block that must be filled with real values. "Looks good" cannot satisfy it.

eval-before-ship — completion statement
Eval complete.
Feature: document-classifier
Suite: evals/classifier/suite_v2.json (120 examples)

Metric: accuracy
Pass rate: 91.7% (threshold: 85%)
Baseline: v1.0.0 @ 88.3% (+3.4%)

Failures: 10 / 120
Pattern: ambiguous category boundary (contract vs. agreement)
Mitigation: added 8 examples to training set — retest pending

Harness: evals/run_eval.py (reproducible)
CI: eval workflow added to .github/workflows/eval.yml ✓

Verdict: PASS — feature cleared for merge ✓

An agent that skips the eval cannot fill in Pass rate: 91.7% and Baseline: v1.0.0 @ 88.3% with real numbers.
That is the enforcement.

The 8 Skills

One skill per failure mode

Three hard gates and five workflow skills. Hard gates block ship if their Completion Statement Format is not satisfied.

eval-before-ship Named eval suite, defined metric, pass rate, failure analysis, and baseline comparison — all five, or no merge Hard Gate
ai-safety-review Prompt injection, hallucination risk, output safety, and agentic scope — four categories, BLOCK verdict required to proceed Hard Gate
fallback-required Timeout, malformed output, low confidence, and refusal — all four failure modes handled before ship Hard Gate
prompt-versioning Semver files, git history, rollback in one line — no prompt edited directly in environment variables Workflow
rag-pipeline-design Data audit and query audit before any pipeline decision. Recall@k baseline required before production Workflow
model-benchmarking Four-stage process: define bar → test set → benchmark three tiers → cost-quality decision. No model picked without task data Workflow
context-optimization Five-level reduction hierarchy — trim before compress, compress before split, split before cache Workflow
ai-cost-audit Token counting, cost projection, and lever table before any optimization claim. No "we could save X%" without numbers Workflow
The 5 Agents

Specialists, not generalists

Each agent enforces one domain. Each has an explicit "What You Don't Do" section — the list of shortcuts it refuses to take.

Sonnet

prompt-engineer

Writes, versions, and iterates prompts. Never deploys without eval-before-ship. Never claims quality without running the eval suite.

Sonnet

eval-designer

Designs eval suites with representative test sets, defines metrics, writes harness code, calibrates LLM-as-judge baselines.

Opus

rag-architect

Runs data audit and query audit before any pipeline design. Never skips the chunking strategy step. Produces recall@k baselines.

Sonnet

model-selector

Four-stage benchmarking process. Never picks a model without task data, never recommends frontier when mid-tier is sufficient.

Opus

ai-safety-reviewer

Reviews all four risk categories. Produces a safety report at safety-reviews/<feature>/<date>.md. Issues are PASS or BLOCK — no middle ground.

What changes when enforcement exists

The difference is not catching issues in code review. It is the work being done before the PR is opened.

✓ Benchmarked 3 tiers. Mid-tier passes the bar at 10× lower cost. Documented.
✓ Data audit done. Recall@8 = 0.87. Chunking strategy justified. Baseline set.
Installation

30 seconds to enforce quality

Sparse-clone installs only the skills and agents directories. Nothing else touches your project.

macOS / Linux
curl -fsSL https://raw.githubusercontent.com/
RBraga01/builder-ai/master/install.sh | bash
Windows (PowerShell)
irm https://raw.githubusercontent.com/
RBraga01/builder-ai/master/install.ps1 | iex
The installer copies skills/ and .claude/agents/ into your project using cp -n (non-destructive — won't overwrite existing files). No global changes, no dependencies installed, no configuration required.
